# ADR-001: Agent Communication Protocol (ACP)

**Status**: Accepted

**Date**: 2026-06-11

**Constitution Reference**: `docs/constitution/acp-v1.md` (ratified 2026-06-06)

## Context

AgentOS agents must exchange messages to spawn children, assign tasks, report
results, request approvals, share memory, and negotiate resources. Without a single
enforced protocol, agents would develop ad-hoc channels -- direct function calls,
shared databases, filesystem polling -- each with its own security model, retry
semantics, and failure modes. The Kernel could not enforce permissions, detect
unauthorized data flows, reconstruct causal chains, or guarantee workspace isolation.

Previous multi-agent frameworks (AutoGen, CrewAI, LangGraph) treat communication as
an implementation detail: Python function calls, shared dicts, or LLM-mediated
conversations with no formal protocol. These lack cryptographic identity (any agent
can impersonate another), ordered delivery (race conditions corrupt state),
backpressure (a slow agent floods the system), and dead letter handling (failed
messages vanish silently).

The ACP constitution mandates: "No agent may communicate outside of ACP. No
component may bypass ACP's signing, routing, or validation."

## Decision

We adopt ACP as defined in `acp-v1.md` as the exclusive communication mechanism.
Key decisions:

### Message Identity and Integrity

Every message carries a UUID v7 identifier and an Ed25519 signature over a
canonical byte sequence: `id|type|channel|sender|recipient|timestamp|SHA-256(payload)`.
UUID v7 enables efficient range queries and natural FIFO ordering. Ed25519 produces
compact 64-byte signatures that verify in microseconds. The canonical form is
deterministic, eliminating signature malleability.

### Routing Modes

Five modes cover every communication pattern:

1. **Direct**: Point-to-point to one AgentID (RPC, task assignments).
2. **Channel**: Pub-sub to all ChannelID subscribers (event streams, audit logs).
3. **Broadcast**: Workspace-wide to all agents (shutdown, configuration).
4. **Topic**: Pattern-matched to subscribers (e.g., `task.*`) for observability.
5. **Priority**: Delivery ordering constraint -- higher priority preempts lower.

Cross-workspace delivery is forbidden unless an explicit bridge exists.

### Delivery Guarantees

- **At-most-once**: Heartbeats, metrics. Retry wasteful when next arrives in 30s.
- **At-least-once**: Task events, notifications. Duplicates tolerated via idempotent
  consumers. Default guarantee.
- **Exactly-once**: RPC and approvals. Implemented via idempotency keys; callee caches
  results keyed by the caller's key. Keys expire after 24 hours.

### Encryption Tiers

- **None**: Default. Signing still mandatory for integrity.
- **E2E**: X25519 + AES-256-GCM between two agents. Kernel cannot decrypt.
- **Channel**: AES-256-GCM with shared symmetric key from channel owner.
- **Workspace**: AES-256-GCM with workspace key from Kernel on entry.

### Priority and Preemption

| Level | Numeric | Can Preempt | Typical Use |
|-------|---------|-------------|-------------|
| CRITICAL | 0 | LOW, BACKGROUND | Error recovery, shutdown |
| HIGH | 1 | LOW, BACKGROUND | Deadlines, user interactions |
| NORMAL | 2 | None | Default agent work |
| LOW | 3 | BACKGROUND | Analytics, logging |
| BACKGROUND | 4 | None | Cleanup, reporting |

CRITICAL messages have unlimited rate limits. `messaging.critical` permission
required to prevent abuse.

### Heartbeat and Liveness

State machine detects failures within 90s:

```
HEALTHY -> SUSPECT (30-60s) -> DEGRADED (60-90s) -> FAILED (90s+)
```

DEGRADED marks tasks "at risk" and notifies the Manager/Chief. FAILED freezes
resources, reassigns tasks, and force-terminates after 5 minutes. Jitter of
+/-5s on 30s intervals avoids thundering herd.

### RPC with Backpressure

Callee returns `error.rate-limit` with `retry_after_ms` when queue is full. After
5 consecutive rate-limits, caller opens a circuit breaker for 60s. Idempotency keys
ensure retried RPCs return cached results without re-execution.

### Dead Letter Queue

Messages failing after max_retries (3 NORMAL, 1 LOW, 0 BACKGROUND) enter the DLQ.
7-day retention. Admin agents can inspect and replay. DLQ is persistent, survives
Kernel restarts.

### Approval Flow

Urgency-based timeout defaults: Critical=1min, High=5min, Normal=30min, Low=4hr.
Escalation to backup approver after a delay. Timeout expires = auto-denied.
Approval results are idempotent.

## Consequences

### Positive

- **Single communication fabric**: Every message flows through one protocol. No side
  channels subvert permissions, isolation, or audit.
- **Cryptographic identity**: Ed25519 signing makes impersonation infeasible. Every
  action is attributable to a verified identity.
- **Auditable by default**: Canonical form + correlation/causation IDs create complete
  distributed traces.
- **Graceful degradation**: Heartbeat state machine, circuit breaker, and DLQ handle
  failures explicitly rather than via silent drops.
- **Flexible security**: Four encryption tiers match effort to sensitivity.

### Negative

- **Signing overhead**: Every message requires Ed25519 sign/verify. At 10K
  msg/s, measurable CPU cost. Batched verification may be needed.
- **Protocol rigidity**: Every interaction must be a message type. Quick-and-dirty
  integrations (shared files, local calls) are forbidden.
- **DLQ operational burden**: 7-day retention requires monitoring. Automated replay
  policies may be needed for high-volume incidents.
- **Key management**: E2E requires X25519 keypairs per agent. Channel/workspace
  encryption requires key distribution. Rotation adds complexity.

### Risks and Mitigations

- **Performance**: Batch-verify at channel level or offload to co-processor.
- **Protocol evolution**: Version field locked to "1.0". Future versions coexist
  via version negotiation during agent registration.
- **Message size limits**: 1MB total / 512KB payload. Large artifacts use shared
  storage with URI references in the payload.
